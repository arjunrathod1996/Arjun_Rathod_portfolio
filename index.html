// DB Trigger Audit Trail — How It Works

MySQL AFTER INSERT / AFTER UPDATE / AFTER DELETE triggers are defined on critical tables (user, lead, contact, activity, etc.). Every change is recorded in the sec.audit_log table with: table_name, operation_type (INSERT / UPDATE / DELETE), row_id, old_data (JSON), new_data (JSON), and changed_by. This provides full traceability — you can see exactly what changed, who changed it, and what the before/after values were.

DB Trigger · Audit Log Record 1

UPDATE Operation — Before & After Values

Audit record shows: Table: user · Operation: UPDATE · Row ID: 6 · Old data: phone_number "8095075541" · New data: phone_number "8095075540" · Changed by: root@localhost. Full field-level change tracking stored as JSON.

DB Trigger · Audit Log Record 2

Sequential Audit Trail — Full History

Record 2/3 shows the reverse UPDATE: Old data: phone_number "8095075540" → New data: phone_number "8095075541". All 3 audit records for this row show a complete history of every state change — critical for compliance, debugging, and regulatory requirements.
